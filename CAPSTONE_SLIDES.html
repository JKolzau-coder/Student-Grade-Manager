1 · Problem: Account Management & Registration

Without the systemWith the system
Duplicate student IDs — no uniqueness checkhasStudent() guard rejects duplicates
Grades outside valid range corrupt averagesGrade [1.0–5.0] enforced at construction time
No audit trail — who changed what, whenStructured records + git history as audit log
Manual average calculation — arithmetic errorsDeterministic calculateAverage() with assertion
Build failures block developer 15–45 minBuildAgent resolves common failures in <5 min
Registration flow: addStudent(name, studentId) → creates identity record. hasStudent() → uniqueness check. addGrade() → profile enrichment. Structurally identical to user registration — academic data instead of credentials.
IAM gap: Any caller can invoke addStudent with any ID. No authentication. No authorization. This becomes a Spring Security problem at live deployment.

2 · Bug via SpotBugs → Fixed by BuildAgent

The Bug: EI_EXPOSE_REP

// VULNERABLE — caller can do:
// service.getGrades(1001).clear()
public List<Grade> getGrades(int id) {
  return grades.get(id); // direct ref
}

// CORRECT — mutation blocked at call site
public List<Grade> getGrades(int id) {
  List<Grade> l = grades.get(id);
  return l != null
    ? Collections.unmodifiableList(l)
    : Collections.emptyList();
}

BuildAgent Detection Flow

mvn verify fails[ERROR] High: getGrades() EI_EXPOSE_REP
detectGate()"spotbugs-maven-plugin" in output → gate = spotbugs
extractErrors()Collects [ERROR] lines
findAffectedFiles()Regex → GradeServiceImpl.java
callClaude()Reads file, identifies getGrades(), applies fix
retry passesSpotBugs gate green → exit 0
This regression actually happened: A previous BuildAgent commit reverted unmodifiableList() to a direct return. SpotBugs caught it on the next mvn verify. The CI workflow now enforces review structurally: BuildAgent pushes to a fix/<gate> branch and opens a pull request with a review checklist — the merge is impossible to skip.

3 · Architecture — Data Format and Generation

public final class Student {
  private final int    studentId;
  private final String name;

  public Student(String name, int studentId) {
    if (name == null)
      throw new IllegalArgumentException(...);
    this.name      = name;
    this.studentId = studentId;
  }
  // getters only — no setters
}
public final class Grade {
  private static final double MIN = 1.0;
  private static final double MAX = 5.0;

  private final String subject;
  private final double  value;

  public Grade(String subject, double value) {
    if (subject == null) throw ...;
    if (value < MIN || value > MAX) throw ...;
    this.subject = subject;
    this.value   = value;
  }
}
ObjectConsequence
final classNo subclassing → SpotBugs EI_EXPOSE_REP2 eliminated at source
final fields, no settersImmutable after construction — concurrent reads safe without locks
Validation in constructorNo invalid object can ever exist — fail fast, not silently

11 · Security Scanning — Three Layers

Version Test

dev time

Runs as part of mvn test.
Asserts that specific dependency versions meet the minimum safe version.

If the assertion fails, surefire output contains GHSA- or CVE- → BuildWrapper detects gate "osv" → BuildAgent fixes pom.xml.

OSV Scanner

local + CI

Runs after mvn verify succeeds.
Generates a CycloneDX SBOM and checks all dependencies against the GHSA / OSV advisory database.

Exit ≠ 0 → same gate "osv" → BuildAgent upgrades the affected version in pom.xml.

Trivy

CI only

Runs in GitHub Actions after the Docker image is built.
Scans all image layers for CRITICAL / HIGH CVEs — catches transitive vulnerabilities that neither the version test nor OSV Scanner saw.

No BuildAgent integration — CI blocks, human review required.

LayerWhenWhat it checksOn failure
Version Test mvn test — before commit Known CVEs in pinned versions (hardcoded assertions) BuildAgent → upgrades pom.xml
OSV Scanner After mvn verify — local & CI Full SBOM against GHSA / OSV database BuildAgent → upgrades pom.xml
Trivy CI — after Docker build All image layers, transitive dependencies CI blocks — manual fix required

13 · Next Steps

IAM & API Authorization

Security
  • REST API via Spring MVC — controller above GradeService, no impl changes
  • JWT / OAuth2 authentication on every request
  • @PreAuthorize("hasRole('TEACHER')") on interface methods
  • Non-root Docker user, minimal base image, no debug ports

LangChain — Commit Memory

AI
  • Index git log + BuildWrapper output into a vector store
  • Before calling Claude: retrieve similar past failures and their fixes
  • Closes the pattern-tracking gap — BuildWrapper gains persistent memory
  • Recurring patterns auto-proposed as new CLAUDE.md rules

14 · Learnings

Commit & Changelog Quality

  • Automated fixes are only as good as the error context passed to Claude
  • Vague gate output → vague fix
  • Structured prompts with explicit error lines → reliable changes
  • Consistent commit format is a prerequisite for traceability

Spring Security at Docker Level

  • @PreAuthorize on GradeService — no rewrites needed
  • Service-layer security ≠ image-layer security
  • Base image CVEs, exposed ports, runtime user — separate concern
  • Both layers must be addressed before live deployment

Pattern Tracking is Hard

  • BuildWrapper retries 3× but has no memory across sessions
  • Same gate can fail repeatedly in different builds
  • No mechanism to recognise, escalate, or learn from recurring errors
  • Each run starts cold — root causes persist
